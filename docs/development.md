## Руководство для разработчиков FolderSync

Этот документ описывает, как работать с кодовой базой проекта, запускать приложение в режиме разработки и вносить изменения.

---

### 1. Стек и зависимости

- Язык: **Python 3.8+** (рекомендуется 3.10+).
- GUI: стандартная библиотека `tkinter`.
- Логирование: сторонняя библиотека `loguru`.

Установка `loguru`:

```bash
pip install loguru
```

---

### 2. Структура проекта

Проект разделён на модули для улучшения поддерживаемости кода:

- **`main.py`** — точка входа приложения:
  - инициализация логирования (`loguru`);
  - создание экземпляра `ConfigManager`;
  - запуск главного окна (`MainWindow`).

- **`config_manager.py`** — модуль работы с конфигурацией:
  - класс `ConfigManager`: управление списком пар папок, загрузка/сохранение `config.cfg`;
  - функция `parse_config_line()`: парсинг строк конфигурации с поддержкой активности и исключений.

- **`sync_core.py`** — ядро синхронизации файлов:
  - функции `scan_directory()`, `is_excluded()`, `should_copy_file()`: работа с файловой системой;
  - функция `sync_pair()`: основная функция синхронизации одной пары папок;
  - класс `SyncResult`: хранение результатов синхронизации (количество скопированных файлов, ошибки и т.д.).

- **`ui_main.py`** — модуль главного окна:
  - класс `MainWindow`: управление главным интерфейсом приложения;
  - обработка запуска/остановки синхронизации;
  - отображение прогресса и логов в текстовой области.

- **`ui_settings.py`** — модуль окна настроек:
  - класс `SettingsWindow`: управление окном редактирования конфигурации;
  - редактирование списка пар папок с чекбоксами активности;
  - выбор папок через диалоги.

- **Документация:**
  - `README.md` — общее описание проекта, установка и базовое использование.
  - `RULES.md` — правила работы с проектом.
  - `ROADMAP.md` — планы развития.
  - `docs/` — подробная документация:
    - `overview.md` — архитектура;
    - `usage.md` — сценарии использования;
    - `configuration.md` — формат `config.cfg`;
    - `development.md` — этот файл;
    - `NEXT_AI_SESSION.md` — контекст для следующей сессии разработки с ИИ.
  - `CHANGELOG.md` — список изменений (обновлять при значимых правках).

- **Служебные файлы:**
  - `.gitignore`, `.gitattributes` — настройки git.

### 3. Запуск в режиме разработки

1. Клонировать репозиторий и перейти в каталог проекта.
2. Запустить `setup_venv.cmd` для создания venv и установки зависимостей.
3. Запустить приложение:

```bash
venv\Scripts\python.exe main.py
```

Или использовать `run_sync.cmd` / `run_sync_silent.cmd`.

При изменении кода:
- достаточно перезапустить приложение;
- состояние GUI и конфигурации полностью определяется текущим `config.cfg`.

---

### 4. Модульная структура проекта

Проект разделён на модули для улучшения поддерживаемости кода:

- **`config_manager.py`** — работа с конфигурацией:
  - класс `ConfigManager` для управления списком пар папок;
  - функция `parse_config_line()` для парсинга строк конфигурации;
  - загрузка и сохранение `config.cfg`.

- **`sync_core.py`** — ядро синхронизации:
  - функции работы с файловой системой (`scan_directory`, `is_excluded`, `should_copy_file`);
  - функция `sync_pair()` для синхронизации одной пары папок;
  - класс `SyncResult` для хранения результатов.

- **`ui_main.py`** — главное окно:
  - класс `MainWindow` для управления главным интерфейсом;
  - обработка запуска/остановки синхронизации;
  - отображение прогресса и логов.

- **`ui_settings.py`** — окно настроек:
  - класс `SettingsWindow` для управления окном редактирования;
  - редактирование списка пар с чекбоксами активности.

Преимущества модульной структуры:
- проще тестировать ядро синхронизации без запуска GUI;
- легче вносить изменения в интерфейс пользователя, не затрагивая логику работы с файлами;
- меньше риск «сломать всё сразу» при доработках;
- улучшенная читаемость и поддерживаемость кода.

---

### 5. Логирование и отладка

- Логирование настроено через `loguru` (см. верхние строки `main.py`).
- При отладке:
  - при необходимости можно временно поднять/изменить уровень логирования;
  - для сложных мест имеет смысл добавлять отладочные (`logger.debug`) сообщения;
  - при отлове исключений использовать декоратор `@logger.catch` (уже используется для некоторых функций).

Хорошей практикой является:
- не убирать полезные отладочные сообщения после нахождения ошибки, если они помогают диагностике поведения программы в будущем.

---

### 6. Добавление новой функциональности

При добавлении нового функционала:

1. Определите место, куда логически относится изменение:
   - GUI (новые кнопки, окна, элементы управления);
   - конфигурация (формат `config.cfg`, новые параметры);
   - ядро синхронизации (новые правила выбора/копирования файлов);
   - документация.
2. По возможности:
   - вынесите новую логику в отдельную функцию/модуль;
   - избегайте чрезмерного роста размера функций.
3. Обновите документацию:
   - `README.md` (краткое описание новых возможностей);
   - при необходимости один из файлов в `docs/` (детали реализации/использования);
   - при значимых изменениях — **`CHANGELOG.md`**, при необходимости **`ROADMAP.md`** и краткая правка **`docs/NEXT_AI_SESSION.md`**, если меняется ключевая логика или контекст для следующего ИИ.

---

### 7. Тестирование

На текущем этапе автоматические тесты отсутствуют, но рекомендуется:

- по мере выделения ядра из GUI:
  - писать тесты на функции, работающие с файловой системой (создание временных директорий, имитация сценариев копирования/удаления);
  - проверять корректность обработки разных комбинаций размеров и времён изменения файлов.

---

### 8. Стиль коммитов и ветвление

См. файл `RULES.md`. Кратко:
- осмысленные сообщения коммитов на русском;
- для крупных изменений — отдельные ветки;
- изменения в документации — отдельными коммитами или ветками при больших объёмах.
